# Feature: Биллинг и подписки

> Управление подписками, тарифами и оплатами — интегрировано в основное приложение.

---

## Архитектурное решение

**Биллинг реализован как раздел в настройках клиники**, а не отдельный личный кабинет.

### Обоснование

| Критерий | Отдельный портал | Раздел в приложении |
|----------|------------------|---------------------|
| Скорость разработки | Медленнее | Быстрее (MVP) |
| Удобство для 90% клиник | Переключение между приложениями | Всё в одном месте |
| Управление несколькими клиниками | ✓ Удобно | Для Phase 2+ |
| Сложность поддержки | Выше | Ниже |
| Доступ для бухгалтерии | ✓ Отдельный доступ | Делегирование |

**Выбор:** Раздел в приложении для Phase 1-2, миграция на портал при необходимости (Enterprise тариф, сети клиник).

## Доступ к биллингу

### Базовая модель (Phase 1)

```
Настройки клиники
├── Общая информация
├── Сотрудники и роли
├── Подписка и оплата ← Раздел биллинга
│   ├── Текущий тариф
│   ├── История платежей
│   ├── Способы оплаты
│   └── Смена тарифа
└── Интеграции
```

**Доступ:** Только пользователи с `is_owner = 1`

### Делегирование (Phase 2)

Владелец может делегировать доступ другим пользователям через permissions.

```sql
-- Таблица разрешений для делегирования
core_user_permissions
├── user_id
├── clinic_id
├── permission (VIEW_BILLING | MANAGE_BILLING | CHANGE_SUBSCRIPTION)
├── granted_by_user_id       -- кто выдал разрешение
├── granted_at
└── revoked_at (nullable)

-- Проверка доступа к биллингу
SELECT EXISTS(
  SELECT 1
  FROM core_users
  WHERE id = :user_id
    AND clinic_id = :clinic_id
    AND (
      is_owner = 1
      OR EXISTS(
        SELECT 1
        FROM core_user_permissions
        WHERE user_id = :user_id
          AND clinic_id = :clinic_id
          AND permission IN ('VIEW_BILLING', 'MANAGE_BILLING')
          AND revoked_at IS NULL
      )
    )
) AS has_billing_access;
```

### Уровни доступа

| Роль | Просмотр тарифа | Просмотр истории | Смена тарифа | Смена платёжных данных |
|------|----------------|------------------|--------------|------------------------|
| Owner | ✓ | ✓ | ✓ | ✓ |
| VIEW_BILLING | ✓ | ✓ | ✗ | ✗ |
| MANAGE_BILLING | ✓ | ✓ | ✓ | ✓ |

## Feature Flags и платные дополнения (Add-ons)

### Концепция двухуровневого контроля доступа

```
Менеджер Orisios (Админка)
  ↓
Включает/выключает фичи для клиники (за деньги)
  ↓
Владелец клиники (Основное приложение)
  ↓
Раздаёт права пользователям внутри клиники
  ↓
Пользователь получает доступ к функционалу
```

**Примеры платных дополнений:**
- Виджет онлайн-записи (+500 ₽/мес)
- Ортодонтическая карта (+800 ₽/мес)
- Мобильное приложение для пациентов (+300 ₽/мес)
- AI-ассистент OrisAI (+1,500 ₽/мес)
- Интеграция с 1С (+1,000 ₽/мес)
- SMS-уведомления (по факту, пакеты)

```sql
├── clinic_id

├── granted_by_user_id       -- кто из владельцев выдал право
├── granted_at

-- user_id=10, clinic_id=42, permission="USE_ORTHO_CHART", granted_by_user_id=8
```

### Backend: Middleware для проверки доступа к фиче

```python
def check_feature_access(
    user_id: int,
    clinic_id: int,
    feature_key: str
) -> bool:
    """
    Проверка доступа к фиче:
    1. Фича включена для клиники менеджером Orisios
    2. Пользователю выдано право владельцем клиники
    """
    return db.fetchval("""
        SELECT EXISTS(
            -- Проверка 1: Фича включена для клиники
            SELECT 1
            FROM adm_clinic_features
            WHERE clinic_id = $2
              AND feature_key = $3
              AND enabled = 1
              AND (expires_at IS NULL OR expires_at > NOW())
              -- Проверка 2: Пользователю выдано право
              AND EXISTS(
                SELECT 1
                FROM core_user_permissions
                WHERE user_id = $1
                  AND clinic_id = $2
                  AND permission = CONCAT('USE_', UPPER($3))
                  AND revoked_at IS NULL
              )
        )
    """, user_id, clinic_id, feature_key)

# Использование в endpoint
@app.get("/api/v1/orthodontics/chart/:patient_id")
async def get_ortho_chart(patient_id: int, user: User = Depends(get_current_user)):
    # Проверка доступа к ортодонтической карте
    if not check_feature_access(user.id, user.clinic_id, "ortho_chart"):
        raise HTTPException(
            status_code=403,
            detail="У вас нет доступа к ортодонтической карте. Обратитесь к владельцу клиники."
        )

    # ... основная логика
```

### Frontend: Условный рендеринг

```typescript
// hooks/useFeature.ts
export function useFeature(featureKey: string): boolean {
  const { user } = useAuth();
  return user.permissions.includes(`USE_${featureKey.toUpperCase()}`);
}

// Использование в компоненте
function PatientCard({ patient }) {
  const hasOrthoChart = useFeature('ortho_chart');

  return (

      {hasOrthoChart && (
        <Tab label="Ортодонтия">
          <OrthodonticChart patientId={patient.id} />
        </Tab>
      )}
    </div>
  );
}
```

### Типы платных фич

| Категория | Примеры | Уровень доступа |
|-----------|---------|----------------|
| **Интеграции** | Виджет записи, 1С, Email-маркетинг | Отдельные пользователи |
| **Модули** | Ортодонтическая карта, Имплантология | Отдельные пользователи |
| **AI** | OrisAI ассистент | Все пользователи или отдельные |

### Автоматические фичи vs Контролируемые

**Автоматические** (не требуют выдачи прав пользователям):
- Мобильное приложение для пациентов — работает сразу после активации
- SMS-уведомления — используется всеми автоматически

**Контролируемые** (владелец выдаёт права):
- Ортодонтическая карта — выдаётся только ортодонтам
- Виджет записи — выдаётся администраторам
- OrisAI — может быть выдан всем или только администраторам

```sql
-- Автоматическая фича (не требует прав)
adm_available_features
├── feature_key = "patient_app"
├── requires_user_permission = 0  -- не требует явного разрешения

-- Контролируемая фича (требует прав)
adm_available_features
├── feature_key = "ortho_chart"
├── requires_user_permission = 1  -- требует явного разрешения от владельца
```

## Бизнес-логика

### Триальный период

```
Регистрация клиники
  ↓
status = trial
trial_ends_at = +14 дней
subscription_plan_id = NULL (или базовый Solo)
  ↓
По истечении 14 дней
  ↓
Требуется выбор тарифа + добавление карты
  ↓
status = active
```

### Автопродление подписки

```sql
-- Задача выполняется 1 раз в день (ARQ/Celery)
-- Находит клиники, которым пора выставить счёт

SELECT id, clinic_id, subscription_plan_id
FROM adm_clinics
WHERE status = 'active'
  AND (
    -- Последний счёт был выставлен > 30 дней назад
    NOT EXISTS(
      SELECT 1 FROM adm_invoices
      WHERE clinic_id = adm_clinics.id
        AND issued_at > NOW() - INTERVAL '30 days'
    )
  );

-- Для каждой клиники:
-- 1. Создать счёт (adm_invoices)
-- 2. Попытаться списать с карты (через Stripe API)
-- 3. Создать запись в adm_payments
-- 4. Если оплата прошла → status=paid, paid_at=NOW()
-- 5. Если оплата не прошла → status=overdue, отправить уведомление владельцу
```

### Смена тарифа

```
Клиент выбирает новый тариф
  ↓
Если новый тариф дороже (Upgrade)
  ↓
  • Немедленно выставить пропорциональный счёт
  • Пересчитать следующее списание
  • Активировать новые фичи
  ↓
Если новый тариф дешевле (Downgrade)
  ↓
  • Применить с начала следующего периода
  • Уведомить о дате смены
```

## Безопасность

### Хранение платёжных данных

- **НЕ храним** полные номера карт
- **НЕ храним** CVV
- **Храним** только:
  - `card_last4` (последние 4 цифры)
  - `card_brand` (Visa, Mastercard)
  - `expiry_month`, `expiry_year`
  - `external_id` (токен в Stripe)

### PCI DSS Compliance

- Вся обработка карт через Stripe/YooKassa (PCI DSS Level 1)
- Наше приложение использует токены, а не сырые данные карт
- Stripe Checkout / Elements для ввода данных карты

### RBAC для биллинга

```python
def check_billing_access(user_id: int, clinic_id: int) -> bool:
    # Проверка: владелец или делегированный доступ
    return db.fetchval("""
        SELECT EXISTS(
            SELECT 1
            FROM core_users
            WHERE id = $1
              AND clinic_id = $2
              AND (
                is_owner = 1
                OR EXISTS(
                    SELECT 1
                    FROM core_user_permissions
                    WHERE user_id = $1
                      AND clinic_id = $2
                      AND permission IN ('VIEW_BILLING', 'MANAGE_BILLING')
                      AND revoked_at IS NULL
                )
              )
        )
    """, user_id, clinic_id)
```
